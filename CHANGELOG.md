- **GitHub Codespaces support (`.devcontainer/devcontainer.json` + `Makefile` `codespace-setup` target + README §3.4 + docs/env-setup.md §4)**: zero-click cloud dev path added as the 4th `README` §3 "Quick start" option, alongside genai-vanilla jupyterhub (§3.1), local Docker (§3.2), and local venv (§3.3). Motivation: the existing 3 paths each cost ~10-15 min of first-time setup on a new machine (Docker pulls + `git submodule update --init --recursive` for `vendor/genai-vanilla` + dependency installs + `make nlp-assets` for spaCy `en_core_web_sm` + NLTK `vader_lexicon`), and §3.1 has the recurring `nnx-pytorch[lm]` → `thekaveh-nnx[lm]==0.2.0` image-bake-drift follow-up as a long-running pain. Base image: `mcr.microsoft.com/devcontainers/python:3.11-bookworm` (matches the `.python-version` + CI `python-version: "3.11"` pin). VS Code extensions preinstalled: `ms-python.python`, `ms-toolsai.jupyter` + `jupyter-cell-tags` (makes papermill `parameters` tag visible) + `jupyter-keymap` + `jupyter-renderers`. - `notebooks/self_supervised-fmnist-jepa-pytorch/` — new Tier-A single-notebook task. First in-repo demo of the nnx megamerge I-JEPA self-supervised stack: `ViTNN(image_size=28, patch_size=4, in_channels=1, d_model=64, n_layers=2, n_heads=4)` context encoder (~103k params); `build_target_encoder(model.net)` for the frozen EMA target; `JEPAPredictor(embed_dim, n_patches, predictor_dim=32, n_layers=2, n_heads=2)` (~30k params) attached to `model.net` so the optimizer picks up its params jointly; `random_block_mask(n_patches, grid_size=7)` for the context/target patch split; `jepa_train_step_factory(target_encoder, predictor, mask_fn, ema_momentum=0.996)` to wire everything together. Phase 2: `LinearProbe(encoder, embed_dim, num_classes)` freezes the pretrained encoder and trains only a `Linear(64 → 10)` head on top of mean-pooled patch embeddings — the standard SSL evaluation. Substituted Fashion-MNIST for the paper's ImageNet / example's CIFAR-10 (already in the collection, CPU-feasible at Tier-A budget). Recorded run (~90 s CPU, 2 JEPA epochs + 5 probe epochs): JEPA pretrain loss 1.0416 → 0.3162; linear-probe val accuracy 70.62% → 74.43% (well above the 10% random baseline — SSL proof-of-life). Heaviest Tier-A notebook in the collection.
- `notebooks/preference_alignment-toy-dpo-pytorch/` — new Tier-A single-notebook task. First in-repo demo of `nnx.dpo_train_step_factory` + `nnx.NNPreferenceDataset` (Direct Preference Optimization, Rafailov et al., 2023). Tiny `TransformerNN` (`d_model=16`, `n_layers=2`, `n_heads=2`); both `ref_model` (frozen) and `policy` (trained) start from the same init. 16 hand-written `(prompt, chosen, rejected)` triplets — cheerful chosen, gloomy rejected. The DPO factory automatically freezes the reference at construction (`requires_grad=False` + `eval()` mode); the nnx test suite enforces bit-for-bit reference invariance. Recorded run (~7 s CPU, 48 iterations over 12 epochs, β=0.1): DPO loss 0.6931 → 0.0039; mean chosen−rejected log-prob gap +2.1434 → +59.6521 (the DPO contract — gap must strictly increase — verified). §6.3 scaling levers (real preference data, β-sweep, DPO variants IPO/KTO).
- `notebooks/sentiment_classification-vader-mlp-pytorch/` — new Tier-A single-notebook task. Rule-based (VADER, NLTK's `SentimentIntensityAnalyzer`) vs neural (spaCy lemmatize → top-100 BoW → 32-hidden MLP) sentiment classification on an embedded 60-review corpus (20 negative / 20 neutral / 20 positive, tiled 4×). Recorded run (~10 s CPU, 60 epochs): VADER 81.25% test accuracy / neural MLP 79.17% — the "hand-tuned lexicons are remarkably competitive baselines" pedagogical point. §6.3 discusses production trade-offs (VADER wins on out-of-distribution, neural wins on matched-domain). Side-by-side confusion-matrix heatmaps. **`nltk` added to `requirements.txt`** (direct dep of this notebook).
- `notebooks/text_classification-agnews-spacy-mlp-pytorch/` — new Tier-A single-notebook task. First NLP-classification task in ml-eng-lab. AG-News-style 4-topic classification (World/Sports/Business/Sci-Tech) on an embedded 80-headline corpus tiled 4× (CI-friendly — no `torchtext.datasets.AG_NEWS` download). Pipeline: spaCy `en_core_web_sm` lemmatize + lowercase + drop stopwords → train-only bag-of-words vocabulary (top-200 tokens, the standard test-leakage footgun is documented) → L2-normalized featurizer → `nnx.NNModel` MLP (200 → 64 → 4) with cross-entropy. §6 renders sklearn `classification_report` + confusion-matrix heatmap. ~13 s on CPU.
- `notebooks/community_detection-karate-louvain-vs-gnn-pyg/` — new Tier-A single-notebook task. Classical (Louvain modularity-maximization via `community.community_louvain.best_partition`) vs GNN-based (GraphSAGE encoder via link-prediction proxy + KMeans on the embeddings) community detection on Zachary's Karate Club. Recorded run (~6 s CPU): Louvain ARI = NMI = 1.000 (perfect — Karate is the poster-child modularity benchmark); GraphSAGE + KMeans ARI 0.155 / NMI 0.429. The §6.3 prose owns this *honestly*: link-prediction is the wrong proxy task for community detection on Karate; better proxies (GRACE / BGRL) explicitly push apart between-community-but-connected pairs. Spring-layout 3-up viz (truth + Louvain + GraphSAGE+KMeans). Sibling to `notebooks/link_prediction-karate-graphsage-pyg/` — same GraphSAGE encoder, different downstream task + different lesson.
- `notebooks/link_prediction-karate-graphsage-pyg/` — new Tier-A single-notebook task. Link-prediction sibling to `notebooks/node_classification-reddit-gnn-pyg/`: same GraphSAGE architecture but used as an *unsupervised edge encoder* on Zachary's Karate Club (34 nodes, 78 undirected edges, identity features). 2-layer `SAGEConv` (34→32→16) encoder + dot-product edge scorer `σ(z_u · z_v)`, trained with BCE on positive observed edges + fresh-resampled negative non-edges per epoch via `torch_geometric.utils.negative_sampling`. Edge splits via `RandomLinkSplit(num_val=0.1, num_test=0.2)`. Recorded run (~5 s CPU, 200 epochs): train BCE 0.7101 → 0.3060, val AUC 0.673 → 0.735, test AUC noisy (~0.43 on the recorded split; the §6.3 prose owns this — the 30-edge test set has variance of ~10 pp). No `nnx.NNModel.train` scaffolding — the loop is ~15 lines and the checkpoint/scheduler infra doesn't pay back at this scale.
- `notebooks/clustering-iris-kmeans-vs-ae-pytorch/` — new Tier-A single-notebook task. Sibling to `notebooks/dim_reduction-iris-autoencoder-pytorch/`: unsupervised KMeans benchmark on raw 4-D iris features vs on a 2-D autoencoder latent. AE is retrained inline (the original plan called for loading from `dim_reduction-...`'s `runs/<best>/`, but `runs/` is gitignored — the inline retrain is CI-safe). Recorded run (~15 s CPU, AE recon loss 0.5460 → 0.0193): KMeans cluster-vs-species ARI 0.716 / NMI 0.742 on raw features → ARI 0.835 / NMI 0.833 on the AE latent (+12 pp ARI, the AE's reconstruction objective has implicitly grouped similar inputs). §6.2 renders the 2×2 truth-vs-KMeans scatter grid on both feature spaces. §6.3 frames KMeans as a *feature-space-quality test*. Labels are used for evaluation only — clustering itself is fully unsupervised.
- `notebooks/moe-fmnist-mixture-of-experts-pytorch/` — new Tier-A single-notebook task. First in-repo demo of `nnx.MoELinear` + `nnx.moe_train_step_factory`. `MoEClassifier` subclasses `FeedFwdNN` and swaps its first hidden Linear for `MoELinear(in=784, out=128, num_experts=4, top_k=2)`. Trained on Fashion-MNIST with `moe_train_step_factory(aux_loss_weight=0.05)` — the Switch-Transformer load-balancing penalty summed across every `MoELinear`. Recorded run (~18 s CPU, 844 iterations over 2 epochs): train loss 2.4078 → 0.3979; aux loss 1.2794 (random init, above the 1.0 uniform-routing floor) → 1.2218 (after training, more balanced). §6 renders the expert-utilization bar chart on a 640-sample probe; §6.3 owns the data-dependent caveat that perfect 25 % per expert isn't the goal — useful specialization can show up as 30/30/20/20. Same `NNDataset`-default-batch-size workaround as the diffusion + transformer tasks (rebuild loader at `batch_size=128`).
- `notebooks/diffusion-mnist-ddpm-pytorch/` — new Tier-A single-notebook task. First in-repo demo of the nnx megamerge diffusion stack: `NoiseSchedulers.LINEAR(T=100)` for the schedule, `DiffusionMLP(input_dim=784, hidden_dims=[256, 256], time_embed_dim=32)` for the denoiser (~477k params), `diffusion_train_step_factory(schedule)` as the `train_step_fn`, `sample(model, schedule, shape=(16, 784))` for the reverse-diffusion sampling loop. The `NNModel` shell holds a placeholder `FeedFwdNN` swapped for the `DiffusionMLP` (`.net` substitution) so the `.train()` scaffolding still composes. Important detail: NNDataset's default batch_size packs the whole 54k train set into one batch — too few noise-level samples per epoch for diffusion to learn — so the notebook rebuilds the train loader at `batch_size=128` (1266 iterations over 3 epochs). Recorded run: noise-prediction loss 1.0102 → 0.9317; 16 sampled digits rendered as a 4×4 grid. Generations are blurry / mode-mixed at this scale + budget — that's the §6.3 "MLP denoiser is the bottleneck, swap for a U-Net for real fidelity" framing. ~19 s on CPU.
- `notebooks/tabular_regression-diabetes-mlp-pytorch/` — new Tier-A single-notebook task. **First regression task in ml-eng-lab.** sklearn `load_diabetes` (442 samples × 10 features, continuous target). Trains 4 models head-to-head: `sklearn.LinearRegression`, `sklearn.KNeighborsRegressor(k=5)`, `nnx` MLP small `[8]`, `nnx` MLP deep `[32, 16]` — all on the same 70/15/15 split. Surfaces a real footgun: `nnx.NNTabularDataset` coerces targets to `torch.long` (it's hard-coded for classification — see its `__post_init__`); regression requires manual `DataLoader` construction with `float32` targets shaped `(N, 1)` to match `output_dim=1` against `Losses.MEAN_SQUARED_ERROR`. Recorded held-out-test R² (n=67): LinearRegression 0.225, MLP small 0.159, MLP deep 0.130, KNN 0.028 — the textbook "linear is hard to beat on small tabular" lesson. §6 predicted-vs-actual + residual plots for all 4 models. ~22 s on CPU.
- `notebooks/dim_reduction-iris-autoencoder-pytorch/` — new Tier-A single-notebook task. PCA baseline vs two autoencoder variants (shallow `[2]`-bottleneck and deeper symmetric `[3, 2, 3]`) on iris, projected to a 2-D latent. First in-repo demo of `nnx.NNModel.train(..., train_step_fn=autoencoder_step)` outside the transformer LM task — `autoencoder_step` implements `MSE(net(X), X)` and ignores the dummy `y`. Both AEs use the `FeedFwdNN(input_dim == output_dim, hidden_dims=...)` trick — no custom `nn.Module` subclass, the bottleneck is the middle Linear. Recorded run (300 epochs, ~18 s CPU): PCA explained-variance ratio [0.83, 0.13], reconstruction loss shallow `[2]` 0.5722 → 0.0142, deeper `[3, 2, 3]` 0.4212 → 0.0492. Held-out test linear-probe accuracy (LogisticRegression on the 2-D latent): PCA 82.61%, AE shallow `[2]` 100.00%, AE deeper `[3, 2, 3]` competitive. §6.3 owns the data-dependent trade-off: at iris scale the extra capacity in `[3, 2, 3]` can overfit the reconstruction and lose species separation. Sibling notebook: `notebooks/clustering-iris-kmeans-vs-ae-pytorch/` (also shipped this PR) does the unsupervised-KMeans counterpart on the same AE recipe; it retrains the AE inline rather than loading this notebook's checkpoint (the `runs/` directory is gitignored so cross-notebook checkpoint loading isn't CI-safe).
- `notebooks/peft-mnist-to-fmnist-dora-vs-lora-pytorch/` — new Tier-A single-notebook task. First in-repo demo of `nnx.apply_lora_to` and `nnx.apply_dora_to` for cross-task adapter fine-tuning. Pretrains a small FFN (`hidden_dims=[128, 64]`, `109,386` total params) on MNIST for 2 epochs, snapshots the state_dict, then spins up three Fashion-MNIST adapt paths from the *same* pretrained base: (a) full fine-tune control (all params trainable, 109,386), (b) LoRA at `r=8`, `alpha=16` (`9,424` trainable, 8.6% of base; wraps 3 Linear layers), (c) DoRA at same rank/alpha (`9,626` trainable, +202 magnitude params vs LoRA). Also exercises `save_lora_weights` / `load_lora_weights` round-trip into a fresh adapter (39.2 KB checkpoint file vs ~440 KB of base weights). §6.2 calls out the budget-dependent trade-off: at the short 2-epoch-pretrain + 1-epoch-adapt schedule the full-fine-tune control still leads on val accuracy; PEFT wins land at longer horizons on well-converged bases. ~25 s on CPU.
- `notebooks/text_generation-tinyshakespeare-transformer-pytorch/` — new Tier-A single-notebook task. First in-repo end-to-end demo of the nnx transformer fork: trains a 256-vocab BPE tokenizer (`nnx.train_bpe` / `NNTokenizerParams`) on an embedded ~7 KB Shakespeare corpus (tiled 8× for loader-size + BPE frequency), tokenizes the corpus into a fixed-length window dataset (`SEQ_LEN=32`, `y = x.roll(-1)` next-token targets), builds a ~114 k-param `TransformerNN` (`d_model=64`, `n_heads=4`, `n_layers=2`, `max_seq_len=32`, default RoPE + tied embeddings), trains via `model.train(train_step_fn=lm_train_step)` with a custom step that flattens `(B, T, V) → (B*T, V)` for `cross_entropy`, then samples generations via `GenerativeNNModel.generate(prompt, max_new_tokens=32, temperature=0.8, top_k=20, seed=42)` for three Shakespeare-style prompts. At this scale + budget the generations are not coherent — that's the §6.3 "correctness smoke test, not quality" framing, with scaling levers (bigger corpus, bigger model, real TinyStories via HF) called out. ~8 s on CPU.
- `notebooks/knowledge_distillation-mnist-ffnn-pytorch/` — new Tier-A single-notebook task. Born-again training demo (Furlanello et al., 2018) via `nnx.born_again_train`: trains a single-gen FP32 baseline as the reference, then separately runs a 3-generation born-again chain from the same fresh init (gen 0 trains on hard labels; gen k>0 distills from a frozen deepcopy of gen k-1 using `nnx.kd_train_step_factory(teacher=...)`). The recorded run shows the per-generation val-loss trajectory (2.14 → 1.91 → 1.65) and a +21% accuracy delta vs single-gen at the same parameter count, attributable to the soft-label regularization effect. ~25 s on CPU.
- `notebooks/pruning-mnist-ffnn-pytorch/` — new Tier-A single-notebook task. First in-repo demo of `nnx.prune.magnitude_prune`: trains an FP32 FFN baseline (`hidden_dims=[256, 128]`), then deep-copies + prunes at each target sparsity `s ∈ {0.1, 0.3, 0.5, 0.7, 0.9}`, measuring validation accuracy + actual zero fraction. The §6 Pareto plot shows the textbook accuracy-vs-sparsity curve and discusses its knee. `bake=True` default writes zeros into `.weight` directly so the deployable state-dict keys are unchanged. 2:4 semi-structured sparsity (`nnx.prune.semi_structured_24`) is referenced in §6.3 but not exercised — its torchao kernel requires CUDA + Ampere+. ~15 s on CPU.
- `notebooks/quantization-mnist-ffnn-pytorch/` — new Tier-A single-notebook task. First in-repo demo of `nnx.quantize_int8` (PTQ INT8 weight-only via torchao) + `nnx.qat_train_step_factory` / `nnx.QATLifecycleCallback` with `qat_config="8da4w"` (8-bit dynamic activation + 4-bit weight QAT). Trains FP32 baseline, quantizes it post-hoc (PTQ), separately trains a fresh model end-to-end with the QAT callback, then compares val loss / val accuracy / pickled state-dict size / per-batch CPU forward latency side-by-side. Asserts QAT callback advanced through `is_prepared` + `is_converted` and the converted net contains `Int8DynActInt4WeightLinear` modules. Hidden widths pinned to multiples of 32 (`[128, 64]`) to fit the int4 default groupsize cleanly without `padding_allowed=True`. Notebook runs in ~27 s on CPU.
